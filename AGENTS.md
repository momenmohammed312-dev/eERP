## Architecture (Clean Architecture, DDD-flavored)

```text
lib/src/
├── presentation/{domain}/         # screens, widgets, riverpod providers
├── application/{domain}/use_cases/ # one class per operation
├── domain/{domain}/
│   ├── entities/                  # pure Dart, no Flutter/Drift imports
│   └── repositories/              # abstract interfaces only
├── infrastructure/
│   ├── database/                  # Drift tables + DAOs
│   ├── repositories/              # impl of domain interfaces
│   └── devices/                   # fingerprint adapter
├── routing/
├── services/
└── theme/
```

Domains: `person`, `academic` (program/course/class/enrollment), `attendance`, `finance`, `wallet`, `canteen`, `library` (books/files), `settings`.

**Rules**

- Dependency direction: Presentation → Application → Domain ← Infrastructure. Domain depends on nothing.
- `domain/**`: zero imports of `flutter`, `drift`, `riverpod`. Pure Dart only.
- UI calls Use Cases (Application layer), never Repository impls directly.
- No business logic in widget `build()` methods.
- One Use Case = one operation, verb+noun name (`EnrollStudentUseCase`, not `StudentService`).

## State management (riverpod)

- Use `ConsumerWidget` / `ConsumerStatefulWidget` where reactive state is needed.
- Providers live next to features (see overlay paths under `presentation/providers/` or equivalent).
- `ref.watch` inside `build()`; `ref.read` inside callbacks and one-off actions — never `ref.read` in `build()`.
- Do not nest a second `ProviderScope` — one at app root is already configured.
- This project does **not** use `@riverpod` / `riverpod_generator` — define providers manually.






---

## Navigation

- All routes are defined in `lib/src/routing/app_router.dart` — do not scatter route tables elsewhere.
- Prefer typed/named routes from the central config — avoid hard-coded path strings in widgets when a named route exists.
- `context.go()` replaces the stack; `context.push()` pushes on top.
- Redirects and guards belong in the router configuration, not inside individual screens.




---

## Backend — none (local-only)

- No HTTP, no Dio, no remote auth. Remove/ignore `dio_service.dart`, `auth_repository*` network flow.
- Login = local check against `person`/`employee` table in Drift, not network auth.
- All persistence via Drift → `app.db` single SQLite file.
- Daily backup job copies `app.db` to separate folder (see technical report, section on backup).

---

## Fingerprint device

- Adapter Pattern. Interface: `IFingerprintDevice { connect(), getLogs(), onPunch(callback) }`.
- Concrete adapter lives in `infrastructure/devices/`. Swappable per device model (USB or LAN) without touching Application/Domain.
- Do not hardcode device SDK calls outside `infrastructure/devices/`.



---

## Services & shared conventions

- Services are singletons with `ClassName.instance`, returning `FutureEither<T>` via `runTask()`.
- Never pass `BuildContext` into services — use `rootContext` from the global navigator helper when UI is required.
- Logging: `AppLogger`; user feedback: `showGlobalToast()`; dialogs: `showAppDialog()` / `context.showAppDialog()`.
- Primary import barrel: `package:edu_erp/src/imports/imports.dart`.
- File names: `snake_case.dart`; classes: `PascalCase`; private members: `_camelCase`.
- Prefer `const` constructors where possible; avoid `dynamic` without justification.
- No empty `catch` blocks — log, map to failure, or rethrow.
- Do not add packages that duplicate existing stack choices (routing, state, backend).

### File & function size (hard rule for AI agents)

- `.dart` file: **max 70 lines**. Exceeds → split into multiple files (extract widget, extract use case, extract DAO).
- Function/method doing >1 responsibility → extract into named private/helper function.
- Large widget → break into sub-widgets, each own file under same feature folder.
- Never justify going over 70 lines with "it's cohesive" — split anyway, use barrel file if needed for imports.

### Dart / Flutter anti-patterns

- No business logic in widget `build()` methods.
- No direct DB/Drift calls from presentation widgets — go through Use Case.
- No network calls anywhere (scope is local-only).
- Do not use `ref.read` inside `build()`.

## How to add a new domain/feature

1. `domain/{name}/entities/` — define Entity (pure Dart), `domain/{name}/repositories/` — abstract Repository interface.
2. `application/{name}/use_cases/` — one Use Case class per operation.
3. `infrastructure/database/` — Drift table for the entity; `infrastructure/repositories/` — Repository impl.
4. `presentation/{name}/` — screens, widgets, providers.
5. Register route in `lib/src/routing/app_router.dart`.
6. Keep every new file ≤70 lines (see hard rule above).
